# Repository Guidelines

## Project Structure & Module Organization
- `content/` holds the site content as Markdown pages with YAML front matter. Most sections use page bundles like `content/project/<slug>/index.md`.
- `config/_default/` contains site configuration in YAML; `hugoblox.yaml` and `theme.toml` provide theme/module settings.
- `layouts/` stores local template overrides; `assets/` is for pipeline assets (Tailwind, icons, processed images); `static/` is copied verbatim to the site root.
- `data/` includes structured data files; `resources/` and `public/` are generated by Hugo (treat as build output).
- `go.mod`/`go.sum` manage Hugo modules; `package.json`/`pnpm-lock.yaml` manage Node tooling.

## Build, Test, and Development Commands
- `pnpm install` installs Node dependencies for Tailwind and Pagefind.
- `pnpm dev` runs `hugo server --disableFastRender` and serves the site at `http://localhost:1313/`.
- `pnpm build` builds the static site (`public/`) and then runs Pagefind search indexing.
- `pnpm run pagefind` re-indexes search after a build.
- If you prefer Hugo directly: `hugo server --disableFastRender` (dev) or `hugo --minify` (production build).

## Coding Style & Naming Conventions
- Use YAML front matter (`---`) with 2-space indentation in Markdown content.
- Prefer kebab-case names for folders and files (e.g., `content/project/my-project/index.md`).
- Keep dates in `YYYY-MM-DD` format and use Hugo’s `weight`/`date` fields for ordering.
- Avoid editing generated files in `public/`, `resources/`, or `hugo_stats.json` by hand.

## Testing Guidelines
- No automated test suite is configured. Validate changes by running `pnpm dev` and doing a quick visual pass, then `pnpm build` to ensure a clean production build.

## Commit & Pull Request Guidelines
- Recent history uses short, lowercase, present-tense messages (e.g., “update …”, “add …”). Keep commits focused and descriptive.
- PRs should include a brief summary, list affected sections/pages, and add screenshots for visual changes. Link issues when applicable.

## Configuration & Deployment Notes
- `netlify.toml` contains deployment settings; update it only when build commands or output paths change.
- Keep `public/` in sync with the intended deployment flow for this repo.
